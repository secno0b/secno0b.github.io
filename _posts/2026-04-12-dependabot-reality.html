That last number is worth sitting with. Over two thousand packages that hadn't been touched in three years or more. That is not technical debt. That is an archaeology project. And none of it was visible on the Dependabot dashboard. ## The paradox nobody warns you about Here's the part that isn't intuitive. When you start remediating the easy packages first (minor version bumps, dev tooling, anything that won't break a build), your average staleness goes *up*. Why? Because you're removing the low-staleness items from the denominator. The packages that remain are the hard ones: deeply embedded, tightly coupled, nobody-wants-to-touch-them dependencies. Your dashboard looks worse even though your actual risk is going down.
If you're only tracking one metric, you're going to get the wrong signal. You need two: total outdated count and critical exposure count. One measures progress. The other measures risk.
## Fix packages, not repos The traditional model is repo-by-repo remediation. Each team "owns" their repos, they update their dependencies, they move on. This is backwards. When typescript shows up outdated in 47 repositories and eslint in 38, fixing them repo-by-repo means 47 separate teams making 47 separate decisions about the same package. Different versions, different timelines, different levels of testing.

# Fix once. Test once. Roll out everywhere.
Instead, identify the top recurring packages and fix them in batches. One package upgrade, tested once, rolled out everywhere. We found that targeting the top 10 recurring packages eliminated more risk than individually remediating 50 repositories. ## What the tracker actually looks like We built a weekly tracker that pulls scorecard data and GitHub package manifests, enriches each package with current version info from the registry, calculates week-over-week deltas, and logs snapshots over time. The output isn't "how many repos have a good grade." It's which packages are the most outdated, which ones appear the most often, and who owns them. 
$ python ted_priority_tracker.py --week 2026-W15

Processing 44 apps, 230 repos...
Enriching version data for 6,093 packages...

  Total outdated:     6,093 -> 5,847  (-246)
  Critical (>1000d):  2,014 -> 1,998  (-16)

Top priority: log4j-core (22 repos, 1,430 days stale)
Snapshot saved: snapshots/2026-W15.json
The grade distribution is interesting but it's not the number that matters. The number that matters is critical exposure: packages so far behind that they represent real, unpatched vulnerability surface. That number moves slowly, and that's what leadership should be watching. ## The takeaway A clean Dependabot dashboard is not a security posture. It's a checkbox. If your remediation strategy is "each team handles their own repos," you're doing the same work N times and missing the packages that matter most. Count the packages. Measure the staleness. Fix the ones that show up everywhere. Everything else is theater.
